T-SQL Tuesday #200 — Two Red Flags That Ruin HR Data Queries

SQL

This month marks a massive milestone: T-SQL Tuesday #200! The #TSQL2SDAY invitation comes from Brent Ozar who asks us to look at the code we inherit and finish this sentence: “When I’m looking at a query, I bet it’s bad if I see ____.” The T-SQL Tuesday is a monthly blogging event that was created by Adam Machanic (blog | blog) and is maintained by Steve Jones (blog | twitter).

Coming from a background in HR and Payroll data, I have seen many messy queries. When you work with payroll data, employee records, and data migrations, accuracy and speed are everything.

When I open an inherited script, there are two specific things that immediately make me say, “Oh no, I am going to have to rewrite this.”

The first is SELECT *. The second is aliases that make the code unreadable. Both of these are common. Both of them look small at first. But both can create extra work, confusion, and frustration when someone else has to support the query later. I like SQL that is clear. I want to open a query and understand what it is doing without guessing. If I need to stop and decode everything first, that is not what I wanted to do.

Red Flag 1: Seeing SELECT * in Production Code

When I am looking at a query, I bet it’s bad if I see SELECT * instead of a proper list of column names. I know why people use it. It is quick, easy, and convenient, especially when you are testing. But in production code, it usually creates more problems than it solves.

Why it is bad

HR and Payroll tables are often huge. An Employee table might have 100 columns, including system flags, audit dates, and legacy data. If a query uses SELECT *, the database has to pull all 100 columns, even if your Power BI dashboard only needs three of them. This wastes memory and slows everything down.

Also, if someone adds a new column to the underlying table later, SELECT * will push that new column into your data pipeline, which can easily break downstream automated processes.

The Better Way

Always name your columns. It makes the code self-documenting and future-proof.

SQL — Bad
SELECT *
FROM employeepayroll;
SQL — Good
SELECT
    id as employee_id,
    BaseSalaryYearly,
    TaxCode,
    FTE
FROM employeepayroll;

This version is better because it is explicit. I can see what the query needs, and I do not get extra columns that nobody asked for.

Red Flag 2: Too Many Aliases Making the Code Unreadable

Another red flag for me is too many aliases that make the code unreadable. I understand why aliases are used, and they are helpful. In joins, they can make a query shorter and cleaner.

Why it is bad

When every table and column is shortened too much, the query becomes hard to follow. Instead of reading the logic, I spend time decoding what a, b, c, x1, x2, and t3 mean. Good SQL should be easy to read, not feel like a puzzle.

The Better Way

Use aliases that are short but clear. Good aliases make the query easier to read, easier to follow, and easier to support later.

SQL — Bad
SELECT a.EmployeeID, b.DepartmentName, c.Salary
FROM HR.Employees a
JOIN HR.Departments b ON a.DepartmentID = b.DepartmentID
JOIN HR.Payroll c ON a.EmployeeID = c.EmployeeID
WHERE a.Status = 'Active';
SQL — Good
SELECT emp.EmployeeID, dept.DepartmentName, pay.Salary
FROM HR.Employees emp
JOIN HR.Departments dept ON emp.DepartmentID = dept.DepartmentID
JOIN HR.Payroll pay ON emp.EmployeeID = pay.EmployeeID
WHERE emp.Status = 'Active';

This is still short, but now it is readable. The aliases actually help instead of getting in the way.

Final Thoughts

For me, bad SQL is not about something dramatic. Sometimes it starts with small choices that make the code harder to read, harder to maintain, and harder to trust. SELECT * hides intent. Bad aliases hide meaning. Neither of these things looks serious at first. But when queries grow, reports break, or another person has to debug the code, those small choices become real problems.
